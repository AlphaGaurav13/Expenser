02 / Relational Architecture

Data Schema & Relational Integrity

Expenser structures data across 5 distinct tables using strict SQLite foreign keys and cascade triggers to prevent database orphans.

  • Groups & User Pivot: A many-to-many relationship (`group_user` table) tracks group members and joining dates.
  • Expenses: Belongs to a group and links to a specific paying user (`paid_by`).
  • Expense Shares: Stores individual user cost-shares, supporting equal, exact, or percentage splits.
  • Settlements: Records cash transfers between a debtor (`paid_by`) and creditor (`paid_to`).
DATABASE SCHEMAS:
users: id, name, email, password_hash
groups: id, name, description, created_by
group_user: group_id, user_id, joined_at
expenses: id, group_id, paid_by, amount, split_type
expense_shares: id, expense_id, user_id, share_amount
settlements: id, group_id, paid_by, paid_to, amount
Relational Schema Slide 3
Speaker Notes
Slide 3: Database & Relationships
"Now let's review our relational model. A primary challenge in expense splitting is handling complex, flexible shares.

To resolve this, we NORMALIZED our schema into five tables. Users and Groups share a many-to-many relationship managed through a pivot table. Every expense is owned by a group and paid by one user. Crucially, the individual split records are moved to an `expense_shares` table. This isolation is what allows Expenser to calculate arbitrary split amounts—whether equal, exact, or percentage-based—without polluting the main expenses table.

Settlements exist as a separate ledger, allowing users to clear balances through peer-to-peer record tracking."

03 / Assets & Pipeline

Bespoke Design & Vite Compilation

Expenser bypasses generic component frameworks, delivering customized HSL CSS rules directly through Vite's production bundler.

  • Dark UI System: Uses variables for background scales, glass opacity borders, and neon-emerald accents.
  • Vite Bundler: Compiles CSS and Javascript source files, outputting production-ready files in `public/build/assets/`.
  • Server Delivery: When compiling for production, removing the temporary `public/hot` file tells Laravel to serve assets statically from the build folder.
Vite Asset Bundler Slide 4
Speaker Notes
Slide 4: CSS Design & Vite Compilation
"Our styling strategy is custom-designed. We did not use standard Tailwind templates or Bootstrap. Instead, we built a bespoke dark CSS framework from scratch using HSL color tokens. This gives us total control over details like borders, glowing elements, and responsive flexboxes.

Vite is used to compile and bundle our assets. When Vite compiles for production, it places the assets in the `public/build` directory and creates a `manifest.json`. During development, Vite creates a `public/hot` file to run a hot-reloading dev server. If that server is stopped, the presence of the stale `hot` file causes assets to fail. Removing the `hot` file instructs Laravel's `@vite` helper to serve the compiled, static CSS and JS files instead."

04 / Login Architecture

Breeze Authentication Lifecycle

Expenser's secure entry-point relies on Laravel Breeze's session authentication backend, wrapped in our custom dark layout.

  • Credential Entry: Secure login form submits the user's email and password to the server.
  • Database Match: Laravel hashes the input and matches it against the stored bcrypt hash in SQLite.
  • Session Cookie: If valid, Laravel starts a secure database session and sends a encrypted cookie to track the user.
  • Middleware Route Protection: Non-authenticated users trying to access dashboard routes are redirected back to the login page.
Login Architecture Slide 5
Speaker Notes
Slide 5: Login Flow & Breeze Setup
"Let's move on to the authentication flow. We implemented Laravel Breeze to handle user login, registration, and password recovery.

We customized Breeze's views with our dark theme. When a user logs in, their credentials are submitted over POST to the `login` route. Laravel hashes the password and verifies it against the hashed record in SQLite. Once validated, it starts a secure session (configured in `.env` to store sessions in the database) and sets a session cookie in the user's browser. This session cookie is verified on every subsequent request."

05 / Security Measures

Active Security & Rate Limiting

Expenser builds security directly into the HTTP pipeline to protect user accounts and prevent brute-force attacks.

  • CSRF Injection: Every login form generates a unique session token to block Cross-Site Request Forgery.
  • Attempt Throttling: Built-in rate limiter restricts users to 5 login attempts per minute, locking out IPs on failure.
  • Password Salting: Passwords are cryptographically salted and hashed using bcrypt (12 cost factor rounds).
  • Auth Guard Middleware: The `auth` middleware protects all routes under `/dashboard` and `/groups`.
// Routes group with auth middleware
Route::middleware(['auth', 'verified'])->group(function () {
    // Authenticated routes
    Route::get('/dashboard', [DashboardController::class, 'index'])->name('dashboard');
    Route::resource('groups', GroupController::class);
});
Security Pipeline Slide 6
Speaker Notes
Slide 6: Login Security & Middleware Details
"Security is integrated directly into our routing pipeline.

First, **CSRF protection** is enforced on all post requests. If a request is submitted without a valid token, Laravel immediately returns a 419 error.

Second, we use **Login Rate Limiting** to prevent brute-force attacks. The system allows a maximum of five login attempts per minute. If exceeded, it locks the account-IP combination for 60 seconds.

Third, passwords are hashed using **Bcrypt** with a work factor of 12.

Finally, we use the `auth` middleware to restrict access to sensitive routes, ensuring that unauthenticated users cannot view group expenses or settle debts."

06 / User Journey

Lifecycle: Registration to Logout

The application ensures a seamless, secure flow as users register, authenticate, view records, and log out.

  • Registration: New users create an account; passwords are hashed and saved.
  • Authentication: Registered users log in; a session is initialized and session cookies are sent.
  • Secure Session: Users browse groups, log expenses, and settle balances under middleware protection.
  • Logout: Destroys the server session, invalidates cookies, and redirects the user back to the landing page.
